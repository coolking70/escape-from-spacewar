> 阶段性质：V0.5.2 是 **验收修复 + 规则冻结** 阶段。**不开发** 战役/科技树/维修/补给/资源系统。

## 2. 修复的问题清单（Bug Fix）

1. **模拟时长计算错误（时间常量）** — 旧代码用 `maxTicks/120` 反推秒数，与真实 tick 速率脱钩。`battleStats.ts` 改用 `tickToSeconds(tick)=tick/30`，并新增规范常量 `TICKS_PER_SECOND=30`；`secondsToTicks` 反向换算。新增单测 23 锁定“时间换算独立于 maxTicks”。
2. **legacyEngine 静默改写规则集** — `rulesets.ts` 旧逻辑把任意 replay 的 `stamp(..., V3)` 硬编码为 v3，导致 `core-v1/v2` 在 UI 上被“偷偷”显示为 v3，且破坏了规则集分发的真实性。改为 `id: version` 保留真实规则集；`resolveRuleset`/`createSimulator` 遇未知规则集直接抛错（无静默回退）。
3. **escaped 被误计为 destroyed** — `battleStats.ts` 旧判定 `combatState==='destroyed' || !sh.alive` 把 `escaped`（`alive=false`）错算成 `destroyed`。改为仅按 `combatState==='destroyed'` 计数与累计损毁价值；`escaped` 仍通过 `getShipPointValue` 计入“保存舰队价值”。
4. **aura 单测队别错误** — `ruleUnitTests.ts` 中 ally 被放到敌方队伍（查找器匹配到 B 队 `standard`），而光环仅队内生效，导致 3 条用例失败。重写为 ally 与光环源同队。
5. **武器/引擎单测缺少 `destroyed` 标志** — `mk` 辅助只设 `hp=0` 未设 `destroyed`，而真实 `recomputeDerivedV4` 用 `!c.destroyed` 过滤，导致 `weaponsDisabled`/`mobilityDisabled` 永不触发、单测失败。修正为 `destroyed: Math.round(100*r)<=0`。
6. **黄金值生成钩子命名错误** — `main.ts` 误绑 `window.generateGoldenReplayValues`，与规格 `window.runGoldenReplayValues` 不符。已修正为规范名称（内部仍委托 `generateGoldenReplayValues`）。

## 4. alive 与 combatState 的准确语义

### 底层布尔 `alive`
- 仅表示“**结构上未摧毁**”。`alive=false` 可能因 `destroyed`（核心/全组件损毁）或 `escaped`（脱战）两种原因之一触发，二者必须靠 `combatState` 区分。
- ⚠️ **绝不可** 仅凭 `!alive` 判定“摧毁”。

### 语义派生函数（`shipFlags.ts`）
| 函数 | 真值条件（要点） |
| --- | --- |
| `isStructurallyAlive(s)` | `s.alive === true`（未摧毁，含脱战/失能） |
| `isPresentOnBattlefield(s)` | `s.alive && s.combatState !== 'escaped'`（仍在战场） |
| `isTargetable(s)` | `s.alive && s.combatState !== 'escaped' && s.combatState !== 'destroyed'`（可用于目标选择） |
| `isCombatCapable(s)` | `s.alive && !s.mobilityDisabled ...`，脱战/损毁/失能均不可战斗（规范实现见 `combatState.ts`，`shipFlags` 再导出以统一） |
| `isActiveInSimulation(s)` | 仍在模拟推进：`alive && combatState !== 'destroyed'`（脱战舰不再步进战斗，但保留终态） |

### `combatState` 优先级（高→低）
```
destroyed  >  escaped  >  disabled  >  retreating  >  critical  >  damaged  >  normal
```
- `destroyed`：核心或全组件损毁（`alive=false`）。
- `escaped`：触发脱战并已越过脱战边界（`alive=false`，但属“保存下来的战力”）。
- `disabled`：武器/引擎/传感器中任一系统全毁（失能，仍存活、仍在场）。
- `retreating`：已发起撤离（机动中）。
- `critical` / `damaged` / `normal`：按剩余结构 HP 比例划分。

### 撤离状态机（V0.5.2 修正）
- 撤离决策一旦触发即记录 `retreatStartedTick`，**不每 tick 重入**（避免事件刷屏）。
- 撤离途中若武器/传感器失能（`weaponsDisabled`/`sensorsDisabled`）**不会中断撤离**——改用 `isRetreatingManeuver(s) = retreatStartedTick!==undefined && !mobilityDisabled` 作为机动/越界判定依据。
- 撤离越界后转 `escaped`；若在撤离前核心被摧毁则转 `destroyed`；若机动系统失能无法撤离且无法战斗，则可能在战斗中判 `disabled`。

## 5. 点数判定价值公式（`getShipPointValue`）

用于**战斗结束判定**与“保存舰队价值”统计。以 `cost = getVariantDef(ship.variant).cost` 为基准：

| `combatState` | 点数价值 | 备注 |
| --- | --- | --- |
| `normal` / `damaged` / `critical` | `100% × cost` | 仍在战斗 |
| `retreating` | `100% × cost` | 仍属在场战力 |
| `escaped` | `100% × cost` | **保存下来的舰队价值**（即便 `alive=false`，不计为损毁） |
| `disabled` | `50% × cost` | 失能，半价值 |
| `destroyed` | `0% × cost` | 损毁，零价值 |
| legacy 无 `combatState` 字段 | `alive ? cost : 0` | 兼容旧录像 |

- `teamCounts` / `decideVictory` 统一使用 `getShipPointValue` 比较两队剩余点数价值。
- `battleStats` 中的 `survivingFleetValue` = 所有未 `destroyed` 舰的 `getShipPointValue` 之和（含 `escaped`）；`pointsDecisionValue` = 决策所用点数价值。

## 9. 已知限制

1. **core-v4 已冻结**：`GOLDEN_EXPECTED` 内嵌基线不得无故改动；任何修改模拟层（伤害/目标/移动/光环/撤离/派生量）的代码都必须保持 16 例指纹稳定，否则 `runGoldenReplayTests()` 直接判失败。
2. **未开发系统（本阶段明确不做）**：战役层、科技树、维修、补给、资源经济。V0.6 才会进入战役层开发。
3. **v1/v2/v3 共用旧引擎**：它们与 v4 行为不等价（无方向命中、无失能、无撤退、光环/无人机语义以旧引擎为准）。黄金指纹相同只说明“同舰队在共享旧引擎下结果一致”，**不**表示 v1/v2/v3 与 v4 等价。
4. **脱战边界**：`escaped` 仅当舰越过脱战边界且 `retreatStartedTick` 已记录才成立；若全队剩余点数过低由撤离引发结束，判定见 `reason=retreat`/`combatDisabled`，不折算为 `destroyed`。
5. **单文件构建**：`build:static` 产出单文件 `static/index.html`（Worker 回退主线程），适合离线分发，但体积较大（754KB）。
6. **确定性边界**：所有随机性来自 seed 派生的 mulberry32 PRNG；`Math.random`/`Date.now`/`performance.now`/rAF delta 一律不进入 sim。渲染层（镜头/选择/特效）不写入 replay，不影响回放结果。
7. **测试范围**：自检只针对 `sim` 层，不比较渲染/镜头/特效；`runAcceptanceTests()` 的耗时仅用于诊断展示，不参与通过判定。
