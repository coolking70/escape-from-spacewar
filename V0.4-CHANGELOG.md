# SpaceWar V0.4 — 舰队构筑 / 改型系统

> 在 V0.3（阵型/战术/回放/确定性）基础上，新增**点数预算**与**舰船改型（loadout）**，
> 让"开战前的编队决策"成为核心玩法。渲染层仅做展示，绝不回写模拟结果。

## 二、新增类型速览

```ts
type ShipClass = 'Fighter' | 'Frigate' | 'Cruiser';          // 基础舰体
type ShipVariant = 'standard'|'interceptor'|'bomber'|'scout'   // Fighter
                |'escort'|'artillery'|'support'             // Frigate
                |'battleship'|'carrier'|'fortress';         // Cruiser

interface VariantMods {            // 纯数据修正（seed 无关，固定表）
  maxSpeedMul; turnRateMul; baseRangeMul;
  coreHpMul; shieldMul; armorHpMul; sensorHpMul;
  weaponDamageMul; weaponCooldownMul;
  classDamageMul: Partial<Record<ShipClass, number>>;   // 对舰种的伤害乘数
  accuracyBonusVs: Partial<Record<ShipClass, number>>;   // 命中加性
  accuracyPenaltyVs: Partial<Record<ShipClass, number>>;  // 命中惩罚
  closeRangePenalty: number;       // 近距(<0.5×射程)命中惩罚
  pointDefense: boolean;
  supportAura?: { type:'sensor'|'shield'; radius; value; targets?: ShipClass[] };
  droneStrike?: { intervalTicks; damage; maxTargets };
}
interface FleetEntry { shipClass: ShipClass; variant: ShipVariant; count: number; }
interface BudgetConfig { mode:'limited'|'unlimited'|'legacy'; limit:number; }
```

## 四、回放格式（Replay Code）规范

Base64URL(JSON)，v0.4 结构：

```jsonc
{
  "v": "0.4",
  "ruleset": "spacewar-core-v3",
  "seed": 123456,
  "budget": { "mode": "limited", "limit": 1000 },
  "teamA": {
    "fleet": [ { "shipClass":"Fighter", "variant":"interceptor", "count":4 }, ... ],
    "formation": "wedge",
    "doctrine": "aggressive"
  },
  "teamB": { "fleet": [...], "formation":"wall", "doctrine":"defensive" }
}
```

**向后兼容矩阵（decodeReplay 自动归一化）**

| 来源 | 识别 | 舰队处理 | 预算 |
|---|---|---|---|
| v0.2 | 无 `ruleset`，`teamA/teamB` 直接是 `{Fighter,Frigate,Cruiser}` 计数 | 每类 → 标准改型 `FleetEntry` | `legacy`（同 unlimited） |
| v0.3 | 有 `ruleset`，`teamX.ships` 对象 | `ships` 对象 → 标准改型 `fleet` 数组 | 缺省 → `legacy` |
| v0.4 | 有 `fleet` 数组 | 原样保留 | 保留 `budget`；缺省 → `legacy` |

> 任意版本解码后 `encodeReplay` 再编码，`fleet`/`variant`/`budget`/`formation`/`doctrine` 均不丢失（已由确定性测试第 3 项覆盖）。

## 五、预算计算

- 每队默认 `limit = DEFAULT_BUDGET_LIMIT = 1000`。
- 单队已用点数 `Σ VARIANTS[variant].cost × count`（见 `shipVariants.fleetCost`）。
- UI 实时显示「已用 / 剩余 / 超预算红字」；**超预算时禁用「开始战斗」**。
- "无限预算"测试开关 → `mode:'unlimited'`，跳过超额校验（仍显示数值）。
- `legacy` 模式来自旧回放（无预算概念），等价于 unlimited。
- 预设（均 ≤1000 点，只改某一队）：均衡舰队 / 舰载机海 / 炮击舰队 / 堡垒防线 / 反大型舰队。

## 六、支援光环（Aura）与无人机（Drone Strike）计算

**光环（Scout 传感器 / Support 护盾）** — 每 tick `computeAuras()` 重算：
- 遍历存活且带 `supportAura` 的舰船，对半径内同队友军累加 `value`（Scout→命中加成 `auraAcc`；Support→护盾恢复 `auraShield`）。
- **叠加上限**：命中加成 `ACC_CAP = 0.15`（封顶 +15%），护盾恢复 `SHIELD_CAP = 2.0`。
- 每 30 tick 发一次 `supportEffect` 事件（供日志/统计，不回写）。
- 应用：`ship.accuracy = clamp(accuracy + auraAcc, 0, 1.5)`；`ship.shieldRegen += auraShield`，随后 `shield = min(maxShield, shield + shieldRegen)`。

**无人机打击（Carrier）** — `maybeDroneStrike()`：
- `ship.droneNextTick` 初始 = `intervalTicks`（90）；当 `tick >= droneNextTick` 触发，并 `droneNextTick += intervalTicks`。
- 目标：敌方存活舰按到自身距离升序，取前 `maxTargets`（3）。
- 伤害：`ds.damage × (0.85 + 0.3 × rng.next())`（**波动完全来自 PRNG，确定性**），经 `dealDamage()` 结算（发 `hit`/`componentDamaged`/`shipDestroyed`，并触发击毁统计）。
- 触发后发 `droneStrike` 事件（含 `targetIds`），渲染层据此画橙向脉冲束。

**点防御（Escort）** — 武器 `role:'pd'`：开火命中时发 `pointDefenseFired`（**仅在命中时**，避免日志/事件刷屏），渲染层画青色短束。

## 七、开发 / 构建 / 运行

```bash
npm install                      # 安装依赖（three / vite / typescript）
npm run dev                     # 本地开发服务器（热更新）
npm run build                   # tsc 类型检查 + vite 生产构建（dist/）
npm run build:static          # 生成零外部依赖的单文件静态站点（static/index.html，~567KB，内联 JS/CSS）
npm run serve:static         # 本地静态预览（http://localhost:4173）
```

> 确定性约束重申：**固定 tick 推进；`Math.random`/`Date.now`/`performance.now`/rAF 的 delta 一律不作为战斗结果依据**；rAF 只决定渲染频率，倍速只决定每帧推进多少 tick。所有随机战斗行为来自 seed 派生的 mulberry32 PRNG。相同配置（含 seed）必然复现相同战斗。

## 八、测试

浏览器控制台（或源码 `src/sim/deterministicTest.ts`）：

```js
runDeterministicTest()   // 全部用例通过输出 "Deterministic test passed…"，否则输出差异明细
runBalanceTest()          // 默认配置跑 20 局：A/B 胜率、平均 tick、平均剩余、平均总伤害
runBalanceTest(cfg, 50)   // 自定义 ReplayConfig 与局数
```

用例覆盖（确定性）：旧 v0.2 / 新 v0.3 / v0.4 编解码；v0.4 `fleet+variant+budget` 往返；同 seed 两次一致；混合改型（bomber/escort/carrier vs interceptor/artillery/fortress）确定性；倍速 1x/2x/4x 一致；跳转(tick=500)再播放一致；多 doctrine 组合确定。

> 说明：默认对称配置（各 ~990 点）平衡性较好，`runBalanceTest` 常见约 35%/35%/30% 且多局逼近 `maxTicks`（双方互歼到残局）。这是**平衡观察**，非缺陷；可调用 `runBalanceTest(customCfg, N)` 对比不同编队强度。

## 九、已知限制

1. **视觉差异轻量化**：改型仅通过少量装饰 Mesh + 主舰体缩放/配色体现，未引入外部模型（遵守"无 glTF/OBJ/贴图"约束）。
2. **光环目标过滤**：`supportAura.targets` 字段已预留，当前 Scout/Support 改型未限定 `targets`（对全部同队生效）。
3. **点防御加成键按舰种**：`accuracyBonusVs` 按 `ShipClass` 键（Fighter 涵盖 bomber/interceptor/scout 等所有 Fighter 改型），未细化到单一 variant。
4. **平衡参数未经大规模调参**：改型数值为首发设定，推荐用 `runBalanceTest` 做对比验证后再微调 `shipVariants.ts` 中的 `cost`/`mods`。
5. **droneStrike/aura 视觉为示意**：脉冲束/光环为轻量 additive 特效，仅作存在感提示，不参与任何数值结算。
6. **静态站体积**：`static/index.html` ~567KB（主要因内联 three.js），属预期；如需更小可改回 `dist/` 多文件分发。
