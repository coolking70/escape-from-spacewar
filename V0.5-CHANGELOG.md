## 二、FleetPreset（舰队方案）数据格式

`FleetPreset` 只描述**单支舰队**，不含 seed / 对方舰队 / 预算上限。

```ts
interface FleetPreset {
  schemaVersion: number;          // 1
  id: string;                     // 本地唯一 ID（localStorage 用，导入/导出时重新生成）
  name: string;
  createdAt: number;              // 导入/导出时归零，由仓库写入真实时间
  updatedAt: number;
  fleet: FleetEntry[];            // { shipClass, variant, count }
  formation: FormationType;       // 'line' | 'wedge' | 'wall' | 'swarm' | 'random'
  doctrine: DoctrineType;         // 'balanced' | 'aggressive' | 'defensive' | 'kite' | 'focusFire' | 'antiCapital' | 'screen'
  notes: string;
}
```

> 该结构**只用于保存/载入/分享单支舰队**，不写入 replay code，不影响战斗结果。

## 三、舰队码（fleet code）格式

`encodeFleet(preset)` 输出一段可分享字符串：JSON → **UTF-8 安全的 base64url**。

```
{ type: "spacewar-fleet", v: "1", name, fleet, formation, doctrine, notes }
```

- 必须带 `type: "spacewar-fleet"`，以此与战斗录像码明确区分。
- 解码 `decodeFleet(code)`：若粘贴的是战斗录像码，则抛出明确提示 *「这是一段战斗录像码，不是舰队方案码」*；若内容无效/无法解析/无有效舰船，也给出对应明确错误。

### 舰队码 vs 录像码（replay code）的区分
| 维度 | 舰队码（fleet code） | 录像码（replay code） |
| --- | --- | --- |
| 标识字段 | `type: "spacewar-fleet"` | 缺省或 `type: "spacewar-battle"`；含 `teamA`/`teamB` |
| 内容 | 单支舰队（fleet + formation + doctrine + notes） | 双方舰队 + seed + budget + ruleset |
| 是否含 seed | 否 | 是（决定确定性结果） |
| 互贴行为 | 当录像码解码 → 报错并提示；当舰队码解码 → 报错并提示 | 当舰队码解码 → 报错并提示 |
| 影响战斗结果 | 否（仅分享配置） | 是（编码即确定的战斗） |

`replay codec` 在解码时若发现 `obj.type === 'spacewar-fleet'` 会直接抛 *「这是一段舰队方案码，不是战斗录像码」*；`fleet codec` 发现像录像码也会反向提示，避免用户误粘贴静默失败。

## 五、战前分析（Pre-battle Analysis）评分规则

入口 `analyzePreBattle(teamA, teamB)`，**纯静态**、完全基于舰队构成 + 改型静态定义（`VARIANTS` 的 `cost` / `tags` / `bars`），**不消耗 PRNG、不读真实时间、不读任何战斗结果、不写 replay**。

- **总分口径**：所有「分数」都是 `(bars × 数量)` 的加权和，便于人工核对。
  - `tendency`（0~1，按数量加权平均）：`speed / range / firepower / defense / support`。
  - `capability`（具备该 tag 的舰船数量）：`antiFighter / antiCapital / pointDefense / sensor / shield / drone`。
  - `totalPoints` = Σ(`variant.cost` × `count`)；`totalShips` = Σ `count`。
- **优势文本**（透明阈值）：
  - `speed ≥ 0.6` → 机动性强；`range ≥ 0.7` → 远程突出；`firepower ≥ 0.7` → 火力强；`defense ≥ 0.7` → 生存强；`support ≥ 0.4` → 有支援体系。
  - 有 `antiCapital` 舰 → 反大舰手段；有 `pd` 舰 → 点防御。
- **短板文本**：缺 `sensor` → 命中被克；缺 `antiFighter` → 易被 Fighter 压制；缺 `antiCapital` → 面对 Cruiser 被动；缺 `pd` → 难拦无人机；`speed<0.4` / `defense<0.4` / `firepower<0.4` → 对应弱点；若选了 `kite`/`aggressive` 但平均速度低，额外提示阵型/战术与速度不匹配。
- **阵型/战术匹配提示**：如 `wall + defensive + (fortress|support)`、`wedge + aggressive`、`swarm + 大量 Fighter`、`wall + antiCapital`、`kite + 低速` 等组合给出匹配/不匹配点评；含 `carrier` 时补充「无人机打击需时间体现」。
- **双方对比提示**（`compare`）：仅在发现明显结构克制时给出，如一方有反大舰而另一方无大舰、一方缺 Scout 而另一方有传感、总点数/舰数显著占优等。
- **免责声明（始终展示）**：`战前分析只比较舰队构成，不代表确定的战斗结果。` 该文案在 UI 顶部以醒目样式呈现。

## 六、Ship Inspector 数据来源

- 数据来源：当前 `BattleState` 中**被选中舰船**的实时快照（位置、各组件 HP / 是否摧毁、存活、当前目标、所属队伍等），由渲染层每帧 `update()` 时读取并刷新面板。
- **只读派生**：Inspector 只显示，绝不写回 `BattleState` / `sim`，因此选择/查看某舰**不会以任何方式改变战斗结果**。
- 与选择联动：点击 3D 舰船 → 高亮 + 相机跟随（可选）+ Inspector 刷新；再次点击空白或同舰取消。

## 七、时间线（Timeline）过滤 / 聚合规则

`simulateFull(replay)` 无渲染地跑完整场战斗得到 `BattleEvent[]`，再经 `buildTimeline` 聚合为关键事件标记（`TimelineMarker`：`tick / type / label / shipId? / team? / importance`）。

- **始终保留**：战斗结束（`battleEnd`）。
- **每种仅取首次**：首次命中（`firstHit`）、首次护盾击穿（`firstShieldDown`）、首个组件被摧毁（`firstComponentDestroyed`）、首舰被摧毁（`firstShipDestroyed`）。
- **10 tick 合并窗口**（`MERGE_WINDOW = 10`）：巡洋舰被摧毁（`cruiserDestroyed`）、无人机打击（`droneStrike`）、大额伤害（`largeDamage`，阈值 `LARGE_DAMAGE_THRESHOLD = 40`）在窗口内合并为单个标记，避免标记过密。
- **忽略的事件类型**（不生成标记，避免噪声）：`weaponFired` / `auraApplied` / `pointDefenseFired` / `supportEffect`。
- **排序**：按 `tick` 升序，同 tick 按类型优先级（`typeRank`）稳定排序。
- **UI 跳转**：HUD 进度条上的标记点可点击，回调 `onSeek(tick)` 触发从初始配置 + 相同 seed 的确定性重模拟到该 tick，再继续播放——**跳转不改变最终战斗结果**（与逐帧播放等价，已写入确定性自检）。
- **重要性着色**：`imp-medium`（橙）/ `imp-high`（红），并以外环区分队伍（`t-a` 青 / `t-b` 品红）。

## 八、平衡实验室（Balance Lab）种子规则 & Worker/主线程 sim 共用

### 种子规则
- 第 `i` 局（0-based）种子：`seed_i = (config.seed + i * config.seedStep) >>> 0`，`i ∈ [0, runs)`。
- `seedStep` 默认 `2654435761`（大素数步长，确保相邻 seed 差异显著；`runBalanceTest` 亦用此值）。
- `maxTicks`：单局最大 tick（默认 4000，可调）；`runs`：局数（10/20/50/100）。
- `swapSides`：若开启，对**每个 seed** 额外以交换双方（teamB 承载 side A、teamA 承载 side B）再跑一局，统计 `sideAWins` / `sideBWins`，计算 `diffPct = |sideAWins - sideBWins| / (2 × runs) × 100`；`diffPct > 5` 触发位置偏差警告（检测是否存在位置相关的不对称）。
- `ruleset`：可选，缺省用当前规则集。
- **统计项**：胜/平局率、平均/min/max tick、平均/min/max 剩余舰船、平均剩余点数、平均伤害、每改型伤害/击毁/损失率/存活/`damagePerCost`/`killsPerCost`、`winner-vs-seed` 列表、长战斗列表（≥90% maxTicks）、零伤害异常、位置偏差。

### Worker ↔ 主线程 sim 共用
- App 优先 `new Worker(new URL('./sim/balanceWorker.ts', import.meta.url), { type: 'module' })` 运行 `runBalance`，通过 `postMessage` 汇报进度与结果。
- 若 Worker 不可用（如 `build:static` 单文件构建无法产出独立 worker 分块），`worker.onerror` 回退到**主线程**直接调用同一个 `runBalance`——**结果完全一致**（纯 sim、同源 `runBalance` / `runHeadless`）。
- 关键：平衡实验室 UI 只读配置、收集参数、展示结果；**不接触任何渲染或运行中的 BattleState**。

### JSON / CSV 导出
`exportBalance(result, format)`：
- **JSON**：`{ type: "spacewar-balance-result", v: "1", ruleset, simVersion, generatedAt, result }`，格式化 2 空格缩进。
- **CSV**：表头 `seed,winner,ticks,teamARemaining,teamBRemaining,teamADamage,teamBDamage`，逐局一行。
- 文件名：`balance-<YYYY-MM-DD>-seed<min>-<max>.{json|csv}`（`min`/`max` 取自 `runsList` 种子的极值）。

## 九、性能优化（大规模舰队）

- **几何共享缓存**：`shipMeshFactory` 维护模块级 `geoCache`（key = `geo.type + JSON.stringify(parameters)`），`acquireGeo` 复用同一几何；`disposeGeometryCache()` 集中释放，避免每艘舰重建几何（50+ 舰船显著降低分配与 GC 压力）。预览页 `shipPreview` 与战斗 `threeScene` 共用该缓存，dispose 不再单舰销毁共享几何。
- **共享叠加几何/材质**：组件损伤标记复用同一 `OctahedronGeometry` + `MeshBasicMaterial`；光环/武器射程/目标连线使用对象池与预分配 `Float32Array`（`targetLinePos`/`targetLineCol` 200×6）。
- **HTML 标签层**：舰名标签走 `labelLayer`（DOM 投影）并复用 `labelPool`，避免每帧创建 DOM。
- **视图筛选纯渲染**：`labels / componentDamage / auraRanges / weaponRanges / targetLines / selectedOnly` 仅影响渲染叠加层，全部在 `threeScene` 内处理，与 sim 完全解耦，可随意开关不重算战斗。
- **确定性重模拟轻量化**：时间线生成走 `simulateFull`（无 Three.js / 无 DOM），不与渲染竞争；跳转 seek 复用同一 `createPRNG + createInitialState + BattleSimulator`。

## 十一、本轮发现的真实缺陷（已修复）

1. **UTF-8 编解码漏洞**：`replayCodec` 与 `fleetPreset` 原先用裸 `btoa(JSON.stringify(...))` / `atob(...)`，对中文舰名/备注会抛 `Invalid character`，导致带中文名的舰队**导出/导入直接崩溃**。已改为「`TextEncoder` → 字节 → base64 → `TextDecoder`」的 UTF-8 安全实现（浏览器与 Node 通用）。
2. **确定性自检误报**：原 `runToEnd`（默认 `maxTicks`）与平衡实验室单局（显式 `maxTicks`）跑了不同长度，造成「单局 vs sim 不一致」的假失败。已在自检中统一 tick 上限后通过——属于测试缺陷而非 sim 缺陷。

## 十二、已知限制

- 平衡实验室的 Web Worker 在 `build:static` 单文件构建下不会产出独立 worker 分块，会自动回退主线程（结果一致，仅占用主线程）。
- 战前分析为**静态构成对比**，不含任何确定性交战模拟，因此不能替代实际战斗结果（UI 已明确免责声明）。
- 舰船选择、相机跟随、视图筛选均为纯 UI/渲染层；它们**不会被写入 replay**，因此分享录像的人看不到你当时的镜头/选择/筛选设置（符合设计）。
- `localStorage` 为单浏览器本地存储，无跨设备/云端同步；导入导出通过舰队码/录像码文本手动分享。
- 大规模舰队（如 50+）在低端设备上帧率可能下降，但**不影响确定性**（tick 结果与舰船数量无关）。
- 静态单文件构建体积较大（~640KB，含 Three.js），这是离线零依赖的代价；开发构建有代码拆分与 worker 分块。
