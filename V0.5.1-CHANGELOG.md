## 二、规则集分发（core-v3 旧规则 / core-v4 新规则）

设计目标：**旧录像码（v0.1~v0.4）继续走旧引擎（当前 `BattleSimulator` = core-v3 行为），绝不偷偷改用新规则；v0.5 录像码走 core-v4 新引擎。**

分发只依据 `replay.ruleset`（解码时已正确映射），**绝不读取渲染/UI 状态**。

```
ReplayConfig.ruleset
   ├─ 'spacewar-core-v1' ┐
   ├─ 'spacewar-core-v2' ├─ RULESETS[...].createInitialState = createStateV3  →  v3 引擎（BattleSimulator）
   ├─ 'spacewar-core-v3' ┘   createSimulator = new BattleSimulator(state, rng)
   └─ 'spacewar-core-v4' ─── RULESETS['spacewar-core-v4'].createInitialState = createInitialStateV4
                            createSimulator = new BattleSimulatorV4(state, rng)
```

- `RULESETS` 是一张 `Record<RulesetId, RulesetDefinition>`；`resolveRuleset(replay)` 只信任 `replay.ruleset`（v4→v4，v1→v1，v2→v2，其余/缺失→v3）。
- 新建战斗默认 `ruleset = spacewar-core-v4`、`v = 0.5`（见 `battleConfig.ts`）。
- 平衡实验室可在 `core-v3` / `core-v4` 间切换，对比同一编队在两个规则集下的胜率/结局差异。
- v1/v2/v3 当前**共用同一份旧引擎代码**，仅在标识上区分；相同 `(config, ruleset, seed)` 在任一规则集下都得到确定结果。

### 旧录像播放规则（不可破坏项）
- 解码时：`v0.1/v0.2 → core-v1`；`v0.5 → core-v4`；`v0.3/v0.4`（含缺省）→ `core-v3`。
- 因此粘贴一段 V0.4 录像码，必定以 core-v3 重算，结果与新规则集彻底隔离——**旧录像的可复现性得到保证**。
- 黄金回归测试同时覆盖 v3（2 例）与 v4（3 例）用例，任何对 sim 层的非预期改动都会使指纹失配而报错。

## 三、方向命中模型（HitZone）

> 文件：`src/sim/componentTargeting.ts`、`src/sim/shipVariants.ts`（`attachHitZones`）
> 纯函数，仅消耗传入 PRNG，且 `selectHitComponent` 的 PRNG 调用次数**恒为 1**（与候选数量无关），保证确定性。

### 3.1 攻击方位判定 `getIncomingHitZone(attackerPos, target)`
与目标朝向一致：前向 `=(cos h, 0, -sin h)`，右舷 `=(sin h, 0, cos h)`。
- 前向投影 `f >= 0.5` → `front`
- `f <= -0.5` → `rear`
- 否则按右舷投影正负：`r >= 0 → right`，`r < 0 → left`

### 3.2 组件候选筛选 + 加权 `buildComponentHitCandidates(ship, zone)`
仅**未摧毁**组件参与；按组件 `index` 升序（稳定、可复现）：
1. **方位过滤**：组件定义了 `hitZones` 且不含当前方位 → 跳过。
2. **护盾/装甲保护降权**：若存在覆盖本方位且保护该组件类型的未摧毁护盾/装甲 → 降权（`shield ×0.5`，`armor ×0.35`）；若其**已摧毁** → 暴露，提权（`×2.2`）。
   - 例：核心 `core` 被左右护盾/装甲保护，左右装甲被摧毁后核心在 `left`/`right` 方位命中率显著升高。
3. 候选按 `index` 升序输出，供确定性加权选择。

### 3.3 确定性选 component `selectHitComponent(cand, rng)`
累加权重，用 `rng.next() * total` 落入区间选唯一组件（PRNG 调用恒 1 次）。方位过滤后无候选时，退回到「全部未摧毁组件按 index 升序」的兜底候选。

### 3.4 伤害类型对装甲倍率 `getDamageMultiplier(dmgType, 'armor')`
| 伤害类型 | 对 armor 倍率 |
| --- | --- |
| `laser` | 0.8 |
| `cannon` | 1.15 |
| `heavy` | 1.3 |
| `pointDefense` | 0.5 |
| `drone` / `kinetic` / 其他 | 1.0 |
其余组件类型倍率恒为 1.0。

### 3.5 各组件命中模型（`attachHitZones` 推导）
| 组件 | 易受攻击方位 `hitZones` | 命中权重 `hitWeight` | 保护 `protects` |
| --- | --- | --- | --- |
| `core` | front/left/right/rear | 0.5 | — |
| `engine` | rear | 1.0 | — |
| `sensor` | front | 0.8 | — |
| `shield` | left/right | 0.9 | `core`（保护核心，未摧毁降权、摧毁暴露） |
| `armor` | 依本地 `offset.x`：左偏→left，右偏→right，居中→全向 | 1.0 | `core` |
| `weapon` | 炮塔→全向；舷炮按 `offset.z` 归 left/right；前向武器→front | 0.9 / 1.0 / 1.0 | — |

> 设计意图：后方攻击更易命中引擎（断机动）；舷侧攻击更易突破护盾/装甲直击核心；核心本身权重低且被护盾/装甲保护，因此**装甲先于核心被摧毁是脆弱化的关键路径**。

## 四、组件摧毁对战斗行为的影响（派生属性）

> 文件：`src/sim/simulatorV4.ts`（`recomputeDerivedV4`）
> 每次受伤后按组件 HP 重新推导，使「打掉特定系统」直接改变该舰作战能力。

| 派生量 | 计算 | 影响 |
| --- | --- | --- |
| `engineRatio` | 未摧毁引擎 HP / 引擎总 HP | `effectiveSpeed = maxSpeed × engineRatio`；`mobilityDisabled = 引擎全毁` |
| `weaponEfficiency` | `0.4 + 0.6 × weaponRatio` | 影响实际伤害与开火冷却（`effCooldown = round(cooldown / weaponEfficiency)`，武器越少冷却越久） |
| `weaponRatio` | 未摧毁武器 HP / 武器总 HP | `weaponsDisabled = 武器全毁`（无法再开火） |
| `sensorRatio` | 未摧毁传感器 HP / 传感器总 HP | `effectiveRange = baseRange × (0.5 + 0.5×sensorRatio)`；`accuracy = clamp(0.5+0.45×sensorRatio, 0.05, 0.95)` |
| `sensorsDisabled` | 传感器全毁 | 命中精度锁定下限、射程减半；接近敌方时视为「失能」触发撤退 |
| `maxShield` / `shieldRegen` | 未摧毁护盾 HP 之和；`regen = maxShield × 0.004` | 护盾是首层吸收，且可被支援光环增益 |

- `mobilityDisabled`（全引擎毁）→ 该舰**无法移动**，但仍可按既有状态被结算（视为失能，不计可战斗单位）。
- `weaponsDisabled`（全武器毁）→ 无法开火，触发撤退判定的「武器系统失效」。
- `sensorsDisabled`（全传感器毁）→ 精度与射程大幅缩水，靠近敌人时被判定为「失能」并促进撤退。
- 这些派生量是**纯数值重算**，无随机、无渲染依赖，不破坏确定性。

## 五、战斗状态机（CombatState）

> 文件：`src/sim/combatState.ts`（`computeCombatState`）
> 优先级（高 → 低，被先命中即返回）：

```
!alive                          → destroyed
escapedTick !== undefined       → escaped
retreatStartedTick !== undefined→ retreating
任一关键系统整体失效（满足其一）  → disabled
   · mobilityDisabled（全引擎毁）
   · weaponsDisabled（全武器毁）
   · sensorsDisabled 且 附近有敌（hasCloseEnemy）
total ≤ 0.25 或 coreRatio ≤ 0.3 → critical
total ≤ 0.6  或 coreRatio ≤ 0.5 → damaged
否则                          → normal
```

- `isCombatCapable`：`normal/damaged/critical/retreating` 视为可战斗；`disabled/escaped/destroyed` 不计。
- 状态转换会推送 `combatStateChanged` 事件；由「全引擎/全武器/全传感器毁」进入 `disabled` 时，额外推送 `mobilityDisabled` / `weaponsDisabled` / `sensorsDisabled` 专项事件供战日志展示。

## 六、撤退 / 脱战（retreating → escaped）

> 文件：`src/sim/retreatSystem.ts`、`src/sim/simulatorV4.ts`

### 6.1 撤退决策 `shouldRetreat(ship, doc, teamLossRatio, rules)`
仅在 `ship.alive && !ship.mobilityDisabled` 时评估，且战术允许（见下）。触发条件（择先命中）：
- 支援/侦察/航母改型：`coreRatio ≤ 0.45 || total ≤ 0.45` → 提前保全（「支援舰保全」）。
- `coreRatio ≤ criticalCoreRatio`（默认 0.2）→ 「核心严重受损」。
- `total ≤ totalHpRatio`（默认 0.18）→ 「舰体濒临损毁」。
- `weaponsDisabled` → 「武器系统失效」。
- `sensorsDisabled` → 「传感器失效」。
- `teamLossRatio ≥ teamLossRatio（默认 0.7）&& total ≤ 0.5` → 「本方重大损失」。

战术倾向（`DEFAULT_RETREAT_RULES`）：
- `defensive` / `kite` 允许撤退；`aggressive` 默认**不允许**（死战）；`balanced/focusFire/antiCapital/screen` 允许。
- `allowDefensiveRetreat` / `allowKiteRetreat` / `allowAggressiveRetreat` 可在运行时调。

### 6.2 撤退移动与脱战判定（关键确定性细节）
- 一旦决定撤退：置 `retreatStartedTick` + `retreatReason` + `combatState='retreating'`，推送 `retreatStarted`。
- **移动**：撤退分支朝「本方出生边界再向外 `ESCAPE_MARGIN+缓冲`」(`escapeTargetX = ±(SPAWN.x + ESCAPE_MARGIN + 6) = ±53`) 移动；**忽略友军分离力**（否则后方友舰分离力会抵消逃离方向，导致卡在原地无法脱战），且 `speedFactor=1.3`（略微提速以脱离追兵）。
- **脱战成功判定在移动之后**：当 `pos.x` 越过本方出生边界外 `ESCAPE_MARGIN`（±47）时 → `escapedTick=tick`、`combatState='escaped'`、`alive=false`、推送 `shipEscaped`、**不触发爆炸**。即「越过边界当 tick 立即脱战」，不会被下一 tick 的火力误杀。
- `escaped` 舰计入对方「retreat」胜利原因；在统计中按改型点数保留价值（`value` 累加 `cost`）。

> 与 `destroyed` 的区别：`destroyed` = 核心毁或全组件毁，触发爆炸、不保留点数价值；`escaped` = 主动撤离战场、存活、保留点数价值（计入 `timeout`/`pointsDecision`/`retreat` 的剩余价值比较）。

## 七、战斗结束判定（VictoryReason）

> 文件：`src/sim/combatState.ts`（`decideVictory`）

当任意一方 `capable == 0` 或达到 `maxTicks` 时调用：

| 条件 | winner | reason |
| --- | --- | --- |
| 双方 `capable==0` 且 `value` 相等 | `null` | `draw` |
| 双方 `capable==0` 且 `value` 不等 | 价值高方 | `pointsDecision` |
| A `capable==0`，B 仍有战斗力 | B | A 全毁→`annihilation`；A 有脱战→`retreat`；否则→`combatDisabled` |
| B `capable==0`，A 仍有战斗力 | A | 同上对称 |
| 双方仍有战斗力（达 `maxTicks`） | 价值高方 / `null` | `timeout` / `draw` |

- `value` = 存活（含脱战 `escaped`）舰船的改型 `cost` 之和；`destroyed` 不计入价值。
- `capable` = 仍 `isCombatCapable` 的舰数（排除 `disabled/escaped/destroyed`）。

## 八、战术深化（doctrine behaviors）

> 文件：`src/sim/simulatorV4.ts`（`targetScoreV4` / `targetInvalid` / `selectTarget` / `anchorFor`）

- **目标评分**（`targetScoreV4`，取最高分）：
  - `balanced`：`-d + (1-r)×6`（兼顾距离与残血）。
  - `aggressive`：`-d×0.5 + (1-r)×45`（极端优先集火残血）。
  - `kite`：`-d`（纯放风筝，远距离优先）。
  - `defensive`：`-d - 0.4×distance(e, centroid)`（靠近本方不冒进）。
  - `antiCapital`：`type权重(Cruiser=2/Frigate=1/Fighter=0)×1000 - d`（专切大舰）。
  - `screen`：Fighter 优先打 Fighter(1000−d)；Cruiser 切大舰；其余贴锚点保护。
  - `focusFire`：团队集火同一最低血目标（`teamFocusTarget`，失效重选最低的）。
- **最大追击距离**（`MAX_CHASE`，超距即放弃当前目标）：`balanced 170 / aggressive 210 / defensive 120 / kite 200 / focusFire 185 / antiCapital 220 / screen 70`。`screen` 极小，保证守在点附近；`aggressive/antiCapital` 追击范围大。
- **锚点（anchor）**：`defensive`/`screen` 战术下的 Cruiser 或 `fortress/carrier/support` 改型标记为 `isAnchor`；`screen` 目标选择时若目标离锚点超过 `protR=70` 则重选（不离开保护半径）；`defensive/screen` 的队形/锚点权重 `ANCHOR_W=0.5`（高于普通 `COH_W=0.18`）。
- **失能目标处理**：`disabled` 目标对**非 aggressive** 视为无效目标（不追打已失能者，符合「失去战斗力即退出交战」的语义）；`escaped/destroyed` 始终无效。
- 所有目标选择按 `ship.id` 稳定排序（等价目标取最小 id），保证确定性。

## 九、轻量转向（movement steering）

> 文件：`src/sim/movementSystem.ts`
> 不使用物理引擎，仅在 XZ 平面合成若干力后归一化。纯函数，无 PRNG、无真实时间。

力构成与权重（来自 V0.5.1 规范）：

| 力 | 权重 | 说明 |
| --- | --- | --- |
| `targetForce` | 1（接近）/ −1（后撤） | 按理想距离带 `desiredBand` 决定接近/后撤/停步 |
| `separation` | `SEP_W=0.9` | 友军排斥，防重叠（按 shipId 稳定顺序累加，避免抖动） |
| `cohesion` | `COH_W=0.18` | 朝本方质心保持结构 |
| `anchor` | `ANCHOR_W=0.5`（defensive/screen）/ `COH_W` | 朝锚点（仅防守体系强） |
| `lateral` | `LAT_W=0.32` | 沿目标切向，确定性 `lateralSign = ship.id % 2 ? -1 : 1`（避免全体同向抖动） |
| `retreat` | 1.5（且忽略 separation） | 撤退时朝出生边界，speedFactor=1.3 |

- **距离带** `desiredBand = 最远武器射程 × 舰种系数(Fighter0.82/Frigate0.9/Cruiser0.98) × 战术系数`（aggressive0.78/defensive1.18/kite1.12/focusFire0.95/antiCapital1.0/screen0.9/balanced1.0）。
- **带内**（0.72~1.08×band）：`speedFactor=0.3` 基本停步，仅做横向+分离微调 → **显著降低抖动/重叠**。
- **太近**（<0.72×band，且 defensive/kite/screen）：后撤 `speedFactor=1`。
- **太远**（>1.08×band）：接近 `speedFactor=1`。
- 合力接近 0 → 原地保持（`speedFactor=0`），避免 NaN/抖动。
- `separation` 半径按舰种：`Fighter 7 / Frigate 11 / Cruiser 17`。

## 十、性能缓存（目标 / 光环）—— 且不影响确定性

> 文件：`src/sim/simulatorV4.ts`

- **目标缓存**：每舰每 **5 tick**（`s.tick - lastTargetEvaluationTick >= 5`）或在目标失效时才完整重选；中间 tick 复用 `ship.targetId`（失效检查 `targetInvalid`：目标死亡/失能/超距/脱离 screen 半径）。缓存只加速选择，**不改变选谁**（重选时评分确定）。
- **光环缓存**：支援光环（`accuracy` / `shieldRegen`）每 **5 tick** 或事件强制（`markAuraDirty`）时重算，存入 `auraAcc` / `auraShield`（按 `ship.id`）。光环计算无随机、按 id 稳定遍历，且对失效源（护盾/传感器已毁）做门控。
- **确定性保证**：缓存仅减少重复遍历，所有涉及战斗结果的量（伤害、精度、速度、目标）均由稳定遍历 + 固定 PRNG 顺序得出；`selectHitComponent` PRNG 调用恒 1 次；等价目标按 `ship.id` 解析。性能优化**不改动 PRNG 调用顺序或 tie-break 顺序**，因此不改变任何结果。
- **v4 出生布局缓存友好**：`applyV4Spawn` 把每队限制在各自半场（A: x∈[−35,−10]，B: x∈[+10,+35]），避免 v3 `wedge` 大舰队跨中线交错（见下），减少出生即重叠，进一步降低分离力抖动。

### v4 出生布局修复（仅 v4，不影响 v3）
- 旧 v3 `wedge` 阵型在大舰队时沿 x 展开可达 66+ 单位，导致双方舰船在出生即交叉（A 前排进入 B 半场），既造成重叠抖动，也使撤退舰需横穿整个竞技场（罕有脱战）。
- `applyV4Spawn`（仅 `createInitialStateV4` 调用，`createStateV3` 字节不变）：
  - `wedge`：按舰种排序后沿 x 前推，但总偏移 `total = min((n-1)×wedgeStep, SPAWN.x − V4_SPAWN_GAP)`，限制在半场内。
  - `swarm`：z 方向按 `swarmScale` 放大，x 不突前。
  - `line/wall/random`：紧凑沿 z 展开，x 固定在出生线（`baseX=±35`），不跨中线。
  - `V4_SPAWN_GAP=10`：最前排距中线至少 10 单位。

## 十五、本轮修复的真实缺陷（V0.5.1 内）

1. **脱战几乎不可见（多因叠加，已逐项修复）**：
   - 撤退转向原指向出生线（≈±35）而非向外脱战边界（±47+），导致舰船被「拉回」无法越界 → 改为 `escapeTargetX=±53`，并在**移动之后**判定越界。
   - 撤退分支原受友军分离力抵消 → 撤退时**忽略 separation**，并提速 `1.3`。
   - `shouldRetreat` 原仅对 `isCombatCapable` 舰评估，漏掉「武器/传感器全毁但可机动」者 → 改为 `ship.alive && !ship.mobilityDisabled`。
   - 根因（v3 既有）：`wedge` 大舰队出生即跨中线交错，撤退舰需横穿全场 → v4 仅以 `applyV4Spawn` 约束半场（v3 字节不变）。修复后实测：1  fortress 巡洋 vs 3 carrier 护卫（seed=1）出现 1 艘因「传感器失效」脱战。
2. **`retreatSystem.ts` 语法错误（缺失 `}`）**：early-return 分支漏括号 → 已补。
3. **`balanceRunner.ts` `ruleset` 未声明**：`const ruleset` 提升到函数顶部，移除循环内重复声明。
4. **`componentTargeting.ts` 字段访问**：`c.type` → `c.def.type`（装甲保护检查）。
5. **`rulesets.ts` 导入**：`PRNG` 从 `battleTypes` 改为 `./prng` 独立导入。
6. **`simulatorV4.ts` 导入**：`createStateV3` 由 `battleState` 具名导入（`createInitialState as createStateV3`）。

## 十六、已知限制

- 平衡实验室的 Web Worker 在 `build:static` 单文件构建下不产出独立 worker 分块，自动回退主线程（结果一致，仅占用主线程）。
- 战前分析为**静态构成对比**，不含确定性交战模拟（UI 已声明）。
- 舰船选择、相机跟随、视图筛选均为纯 UI/渲染层，不写入 replay；分享录像的人看不到你的镜头/选择/设置。
- 大规模舰队（50+）在低端设备帧率可能下降，但**不影响确定性**（tick 结果与舰船数量无关）。
- 静态单文件构建体积较大（~670KB，含 Three.js），是离线零依赖的代价；开发构建有代码拆分与 worker 分块。
- `retreat` 胜利原因仅在败方存在 `escaped` 舰时标记；若败方既全毁又有脱战，优先 `annihilation`（全毁）判定（见第七节表）。
- `scripts/gen-golden.cjs` / `scripts/v4-smoke.cjs` 为开发期一次性工具，依赖 `.tmp-test` 编译产物，不参与发布构建。
